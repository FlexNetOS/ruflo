# LongMemEval Retrieval Benchmark — 2026-05-01

Branch: `bench/longmemeval-2026-05-01`
Runner: `npx tsx benchmarks/longmemeval/run-retrieval-bench.ts`
Index: HNSW (in-process, no quantization)
Dataset: `data/longmemeval_oracle.json` (LongMemEval oracle format)

## Headline

**Smart pipeline (SmartRetrieval / ADR-090) beats raw HNSW** on the metrics that matter for end-user answer quality, at a fixed-k=10 retrieval budget. Wins at n=30 are amplified at n=500.

| Metric | RAW (n=500) | SMART (n=500) | Δ |
|---|---|---|---|
| Session R@10 | 100.0% | 100.0% | 0 |
| Session Top-1 | 100.0% | 100.0% | 0 |
| **Content@1** | 22.2% | **24.6%** | **+2.4 pp** |
| Content@3 | 35.8% | 34.6% | −1.2 pp |
| Content@10 | 43.6% | 43.2% | −0.4 pp |
| **MRR (content)** | 0.2967 | **0.3058** | **+0.0091** |
| Retrieval p50 | 0.02 ms | 5.79 ms | +5.77 ms |
| Retrieval p95 | 0.03 ms | 13.73 ms | +13.70 ms |

The smart pipeline's small p50/p95 increase (still well under 15 ms) is the cost of query expansion + RRF + recency + MMR + session round-robin running over the same HNSW index. Both strategies share embedding + index — the delta is fully attributable to the pipeline.

## Addendum 2026-05-01b — MiniLM (real semantic embedder) vs hash (bag-of-words)

After the runs above, an audit of `run-retrieval-bench.ts` revealed that the previous "embed" function was **not a learned encoder** — it was a deterministic hash of word unigrams + word bigrams + character 3-grams into a 384-dim space. This is bag-of-words, not semantic similarity. ADR-088's roadmap presumed `all-MiniLM-L6-v2` was the existing baseline; in reality there was no semantic encoder in the bench.

The bench was extended with `--embedder hash|minilm|bge-small|bge-base|bge-large` (commit forthcoming). MiniLM (`Xenova/all-MiniLM-L6-v2`, 384d, ONNX-quantized) was run at n=500 for both `raw` and `smart` strategies.

### Headline (n=500, k=10, no API)

| Metric | Hash raw | MiniLM raw | Hash smart | MiniLM smart |
|---|---|---|---|---|
| Session R@10 | 100.0% | 100.0% | 100.0% | 100.0% |
| Session Top-1 | 100.0% | 100.0% | 100.0% | 100.0% |
| **Content@1** | **22.2%** | 17.6% | **24.6%** | 19.4% |
| Content@3 | 35.8% | 32.4% | 34.6% | 31.8% |
| Content@10 | 43.6% | 44.0% | 43.2% | 43.4% |
| **MRR (content)** | **0.2967** | 0.2621 | **0.3058** | 0.2702 |
| Retrieval p50 | 0.02 ms | 0.03 ms | 5.79 ms | 15.52 ms |
| Retrieval p95 | 0.03 ms | 0.08 ms | 13.73 ms | 29.06 ms |

**MiniLM is worse than hash on every Content@k metric except Content@10 (where they tie).** Smart-pipeline delta over raw is consistent across both encoders (~+2 pp C@1), confirming the pipeline contribution is independent of the embedder.

Compared to hash smart, MiniLM smart loses most on `multi-session` (5.3% vs 12.0%, −6.7 pp) and `knowledge-update` (32.0% vs 41.0%, −9.0 pp) — the categories where retrieving the chunk that contains the literal answer string matters most. Where token overlap matters less (`single-session-assistant`, `single-session-preference`), the encoders track each other.

### Why MiniLM lost — and what this means for the optimization roadmap

The bench's `Content@k` metric is **lexical substring match**:

```ts
// run-retrieval-bench.ts:scoreRun()
const matchesAnswer = (content: string): boolean => {
  const c = content.toLowerCase();
  return answerNorms.some(a => c.includes(a));
};
```

A retrieved chunk only counts as a "hit" if the **exact answer string** appears as a substring. Hash's character-3-gram and word-bigram features inadvertently maximize token-overlap — which is what this metric rewards. Dense semantic embeddings (MiniLM, bge-large) optimize for *semantic* similarity, which is a different objective: a chunk that paraphrases the answer ranks high in cosine similarity but scores 0 on substring match.

This invalidates the simplest reading of ADR-088's Tier 1 #1 ("swap MiniLM → bge-large for +5–15 pp"). A stronger semantic encoder optimizes the wrong objective for this metric. **The actual Tier-1 priorities reorder**:

1. **Tier 1 #3 (BM25 + dense via RRF) jumps to first.** BM25 IS lexical retrieval done properly — full-document scoring with TF-IDF and length normalization, not opportunistic token overlap. Combining BM25 (which the metric rewards) with dense (which captures semantic recall when keywords don't match) via RRF should beat both individually.
2. **Tier 1 #2 (semantic chunking) still matters.** Many `single-session-preference` answers (0% across both encoders) are mid-message; chunk boundaries determine whether the answer text lands in retrievable chunks at all.
3. **Tier 1 #1 (bge-large swap) deferred** until after BM25 hybrid lands. May still help on multi-hop categories, but expected gain is smaller than first thought on this lexical-match metric.

There is also a separate, larger question: **is `Content@k` the right metric for AgentDB?** LongMemEval's official evaluator (`evaluate_qa.py`) uses an LLM to grade a *generated answer* against the gold reference — not substring-match the retrieved chunks. The bench measures retrieval-only quality with a strict lexical proxy. Future work: wire `evaluate_qa.py` for an end-to-end (RAG + Haiku QA) score that credits semantic matches.

## Addendum 2026-05-01c — BM25 hybrid (Tier 1 #3) — new SOTA on this bench

The MiniLM addendum surfaced that the bench's `Content@k` metric is strict **lexical substring match**, so the next intervention had to add proper lexical retrieval. Implemented Okapi BM25 + Reciprocal Rank Fusion (Cormack et al. 2009, k_RRF=60) and added a `--retrieval dense|bm25|hybrid` flag to the bench. All hybrid runs do RRF-merge over a fan-out of `max(k*3, 20)` from each retriever.

### Headline (n=500, k=10, no API)

| Config | C@1 | C@3 | C@10 | MRR | p95 |
|---|---|---|---|---|---|
| Hash dense raw (orig) | 22.2% | 35.8% | 43.6% | 0.2967 | 0.03ms |
| Hash dense smart (prev best) | 24.6% | 34.6% | 43.2% | 0.3058 | 13.7ms |
| MiniLM dense raw | 17.6% | 32.4% | 44.0% | 0.2621 | 0.08ms |
| MiniLM dense smart | 19.4% | 31.8% | 43.4% | 0.2702 | 29.06ms |
| BM25 alone raw | 23.4% | 36.0% | 43.8% | 0.3056 | 0.04ms |
| Hybrid hash+bm25 raw | 24.2% | 38.0% | 43.8% | 0.3129 | 0.08ms |
| Hybrid minilm+bm25 raw | 21.8% | 35.6% | 44.6% | 0.2975 | 3.23ms |
| Hybrid minilm+bm25 raw (kRRF=10) | 21.4% | 35.2% | 44.4% | 0.2936 | 3.30ms |
| **Hybrid hash+bm25 SMART** | **26.8%** | 37.0% | 44.0% | **0.3269** | 14.74ms |

**Smart hybrid hash+bm25 is the new SOTA on this benchmark + metric: C@1 26.8% (+2.2 pp over previous best), MRR 0.3269 (+0.0211).**

### Per-category for the new SOTA (smart hybrid hash+bm25)

| Category | n | C@1 | C@3 | C@k | MRR | Δ vs hash smart |
|---|---|---|---|---|---|---|
| knowledge-update | 78 | 41.0% | 62.8% | 70.5% | 0.5254 | tied (was 41.0%) |
| multi-session | 133 | 12.8% | 25.6% | 35.3% | 0.2016 | +0.8 pp |
| single-session-assistant | 56 | **39.3%** | 44.6% | 51.8% | 0.4287 | **+8.9 pp** |
| single-session-preference | 30 | 0.0% | 0.0% | 0.0% | 0.0000 | unchanged (chunking) |
| single-session-user | 70 | **67.1%** | 71.4% | 81.4% | 0.7082 | **+7.1 pp** |
| temporal-reasoning | 133 | 12.0% | 20.3% | 24.1% | 0.1659 | tied |

Big wins where the answer text is anchored in a specific user/assistant turn (`single-session-*` categories). Tied where retrieval isn't the bottleneck (`knowledge-update`, `temporal-reasoning`). Still 0% on `single-session-preference` — the answer text spans message boundaries and never lands in a single retrievable chunk; this is a chunking limit, not a retrieval one.

### Two findings worth keeping

**1. BM25 alone (no smart pipeline, no embeddings) outperforms MiniLM smart.**
BM25 alone raw: C@1=23.4%, MRR=0.3056. MiniLM smart: C@1=19.4%, MRR=0.2702. Even with all the multi-query / RRF / recency / MMR / session-RR machinery in the smart pipeline, MiniLM dense can't beat plain Okapi BM25 on this metric. This is *not* a knock against semantic retrieval generally — it's a result about what the bench measures (lexical substring match).

**2. Hybrid hash+bm25 raw (no smart) is within 0.4 pp of hash smart — almost free.**
Hybrid hash+bm25 raw: C@1=24.2%. Hash smart: 24.6%. The smart pipeline (multi-query expansion + RRF + recency + MMR + session-RR) buys ~+2.4 pp of C@1 over hash dense. Adding BM25 to hash dense (with no smart pipeline) buys ~+2.0 pp. They're complementary: smart on top of hybrid lands at 26.8% — the gains roughly add. The smart pipeline contribution is ~+2.6 pp regardless of the underlying retriever, which makes sense — its query expansion / fusion / ranking improvements are orthogonal to whether the underlying retrieval is sparse, dense, or hybrid.

### Things that didn't work

- **MiniLM in hybrid is worse than hash in hybrid.** Hybrid hash+bm25 raw beats hybrid minilm+bm25 raw on C@1 (24.2% vs 21.8%) and MRR (0.3129 vs 0.2975). MiniLM does win on C@10 (44.6% vs 43.8%) — it surfaces semantically-relevant-but-lexically-weak chunks at deeper ranks — but those chunks rarely contain the literal answer string, so they don't help the substring metric near rank 1.
- **Lowering RRF k doesn't help.** kRRF=10 (top ranks weighted more heavily) on hybrid minilm+bm25 produced 21.4% C@1 vs 21.8% at kRRF=60. RRF over-smoothing isn't the issue; MiniLM's top picks just don't carry lexical weight.

### Implications for the optimization roadmap

- **Tier 1 #3 is done.** BM25 hybrid lands ~+5 pp C@1 over hash dense and 9 pp over MiniLM dense.
- **Tier 1 #1 (bge-large swap) deprioritized.** A stronger semantic encoder might do something here, but the obstacle is the metric, not the encoder. Without changing the metric (e.g., to an LLM-graded answer quality score), bge-large in hybrid likely does about as well as MiniLM in hybrid (i.e., worse than hash+bm25). Worth one ablation later but not the next move.
- **Tier 1 #2 (semantic chunking) is now the highest-leverage open item.** Specifically, `single-session-preference` is 0% across every config tested — the answer never lands in a top-k chunk. Sentence-aware chunking (256-token window, 64-token overlap) would let answer-bearing spans survive the chunk boundary. Expected: +5 pp on `single-session-preference`, smaller gains everywhere.
- **Tier 2 #6 (Haiku multi-query expansion) is the next biggest upside on temporal/multi-hop.** Both `temporal-reasoning` (12%) and `multi-session` (12.8%) C@1 are far below the easier categories' 67% / 39%. The smart pipeline already does multi-query (regex-based paraphrase), but a real LLM paraphrasing the question 3-5 ways would surface different keyword sets BM25 can't reach.
